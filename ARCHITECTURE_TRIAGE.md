## Triage Methodology

Every remaining finding from PROJECT_REAUDIT.md is classified into one of four categories:

| Category | Meaning | Action |
|----------|---------|--------|
| **A — Phase C Blocker** | Will cause data corruption, runtime crashes, or design friction during AI Pipeline development | Fix before Phase C |
| **B — Should Fix Before Release** | Impacts DX, documentation accuracy, or release quality | Fix before v1.0 stable |
| **C — Technical Debt** | Real but safe to postpone; won't block or slow Phase C | Schedule for post-Phase C cleanup |
| **D — Ignore / Won't Fix** | Intentional design, over-engineering, or negligible ROI for a local-first single-user app | Close as "Won't Do" |

## Would I Authorize Phase C After Category A Is Complete?

### **YES — with conditions.**

**The rationale:**

Category A represents the six genuine technical risks that would actively harm Phase C development. Each is either:
- A **data-corruption risk** that schema migrations would trigger (A1),
- A **debugging velocity killer** that would misdirect frontend developers (A2),
- A **security gap** that would widen under programmatic use (A3),
- A **runtime failure** that AI-generated content would trigger at the worst moment (A4),
- An **execution integrity risk** that subprocess-heavy AI code would inherit (A5), or
- A **concurrency freeze** that AI inference calls (far heavier than FFprobe) would make catastrophic (A6).

These six are the right problems to solve before Phase C starts. They are also the *only* ones I would hold Phase C for.

**Why the re-audit's previous "NO" stance was too conservative:**

The re-audit recommended four gates (CI, all High fixes, 95% test pass, accurate README). That would take 1-2 weeks and would mean Phase C starts 1-2 weeks later than necessary, for benefits that are real but deferred: CI doesn't help debug an AI inference pipeline — it helps prevent regressions *after* the pipeline is built. Test pass rate matters more *during* Phase C (as AI code introduces new failures) than at the starting line.

The Architecture Triage separates the six genuine "will hurt during Phase C" items from the 40 items that are real but safe to postpone. Once those six are fixed:

- The **data layer** won't silently corrupt schema migrations (A1)
- The **API layer** will return debuggable errors to the frontend (A2)
- The **upload path** won't write outside expected directories (A3)
- The **FFmpeg layer** won't crash on AI-generated caption text (A4)
- The **subprocess layer** won't resolve from untrusted paths (A5)
- The **event loop** won't freeze on heavy operations (A6)

**The conditions:**

1. Category A must be complete and verified before Phase C work begins.
2. CI/CD (B1) should be added within the first week of Phase C — not a pre-condition, but a very-early priority.
3. The AI dependency validation (B8) must be done before any `pip install` of Phase C dependencies.
4. All Phase C code must use the patterns established by the Sprint 1–3 fixes: `asyncio.to_thread` for blocking calls, `MediaInfo` typed access, chunked streaming, and the corrected encryption/validation utilities.
5. A mid-Phase-C checkpoint review after the first 2 weeks to verify the debt isn't accumulating faster than expected.

**The honest assessment:**

The codebase is at 60/100. After Category A fixes it will be around 64/100 — still not "good" by absolute standards, but adequately de-risked for Phase C to proceed. The concerns that remain (CI, mypy, ruff, test failures, dead code, complexity) are genuine quality issues that should be fixed before v1.0 release, but they are not reasons to delay starting the AI pipeline — they are reasons to schedule a cleanup sprint between Phase C completion and v1.0 release.

**Final answer: YES — authorize Phase C after Category A items A1–A6 are complete and verified.**
